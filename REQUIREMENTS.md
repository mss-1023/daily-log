# 日迹 — 微信小程序需求文档

> 版本 v8（MVP 最终版）· 2026-06-30

---

## 一、项目概述

**日迹**是一个个人记录小程序，涵盖日总结、待办、备忘、心情记录、习惯打卡五大模块，数据全部本地存储，支持 JSON 导入导出永久保存。

**不发布上线**，仅通过微信开发者工具生成预览二维码，微信扫码在手机上使用。

---

## 二、技术决策

| 项目 | 决定 |
|------|------|
| 应用形态 | **微信小程序** |
| 前端框架 | **微信小程序原生**（WXML + WXSS + JS） |
| UI 组件库 | **TDesign 小程序版**（`tdesign-miniprogram`） |
| 数据存储 | **本地 `wx.setStorageSync`**，每种记录独立 key 前缀 |
| 导出格式 | **JSON**（合并全部数据为一个文件） |
| 发布 | **不发布**，仅开发者工具预览 |
| 云开发 | **不使用** |
| 开发工具 | **wechat-web-devtools-linux**（Linux 移植版） |
| 开发系统 | **Ubuntu（Linux）** |

## 三、设计细节确认（开发前必读）

以下 5 个设计细节已在开发前明确，避免实现时反复。

### 3.1 🔥 连续天数计算规则

**仅统计「日总结」（entry_）的连续天数。**

- 从今天向前遍历 `entry_YYYY-MM-DD`，直到某天不存在为止
- 心情、习惯打卡不计入连续天数
- 若今天未写但昨天写了，连续天数显示昨天的值（不中断）
- 在 `streak.js` 中实现 `getStreak()` 方法

### 3.2 📅 是否允许写未来日期

**允许。** 不设限制，用户自主管理。

- 切换到未来日期可正常输入，相当于提前写计划
- 切换到历史日期可补写
- 保存非今日日期时，Toast 提示「已保存 X 月 X 日的记录」
- 首页默认显示今天

### 3.3 📌 首页「今日待办」筛选逻辑

- 仅显示 **deadline ≤ 今天** 的待办
- **无截止日期的待办不显示在首页**，仅出现在待办页（用户若发现添加了待办但首页没出现，是因为没设截止日期）
- 按截止时间从近到远排序
- 最多显示 5 条，超出显示「查看全部 →」
- 0 条时显示「今天没有待办，轻松的一天 ~」

### 3.4 ✅ 习惯补签交互

通过首页箭头切换到历史日期，`habit-check` 组件根据日期读写对应的 `habit_YYYY-MM-DD` key，点击直接打勾/取消。**无需额外页面开发。**

- `habit-check` 组件接收 `date` 参数
- 根据 `date` 读取 `habit_YYYY-MM-DD`，与 `habit_def_*` 列表合并渲染
- 一次读取一个 key，不逐个遍历

### 3.5 📊 月历「绿色圆点」判定标准

检查当天是否存在以下任意一种数据（4 种都检查）：

| 数据类型 | 检查方式 |
|------|------|
| 日总结 | `entry_YYYY-MM-DD` 是否存在 |
| 心情 | `mood_YYYY-MM-DD` 是否存在 |
| 习惯打卡 | `habit_YYYY-MM-DD` 是否存在 |
| 待办 | 遍历 `todo_*` 检查是否有 `deadline` 在当天的 |

一个月最多 31 天，全量检查 4 种数据，开销极小（< 50ms），无需缓存。（待办部分的具体实现方案详见 8.5 分桶优化。）

## 八、存储方案

### 8.1 存储 key 设计

| 前缀 | 内容 | 格式 |
|------|------|------|
| `entry_YYYY-MM-DD` | 日总结 | `{content, tags, created_at, updated_at}` |
| `mood_YYYY-MM-DD` | 心情 | `{emoji}` |
| `habit_YYYY-MM-DD` | 当天习惯完成情况 | `{habit_id: true/false, ...}` |
| `todo_` + id | 待办 | `{title, deadline, note, done, created_at}` |
| `memo_` + id | 备忘 | `{title, content, created_at, updated_at}` |
| `habit_def_` + id | 习惯定义 | `{name, icon, order, created_at}` |

**共 6 种存储模型。**

### 8.2 ID 生成策略

使用 `Date.now().toString(36) + Math.random().toString(36).slice(2, 6)` 生成唯一 ID。

### 8.3 导出文件格式

```json
{
  "version": 1,
  "app_name": "日迹",
  "exported_at": "2026-06-30T22:00:00",
  "entries": [{ "date": "2026-06-30", "content": "...", "tags": ["工作"], "created_at": 1719756000000, "updated_at": 1719756000000 }],
  "todos": [{ "id": "abc123", "title": "提交报销单", "deadline": 1719756000000, "note": "", "done": false, "created_at": 1719756000000 }],
  "memos": [{ "id": "def456", "title": "物业电话", "content": "010-12345678", "created_at": 1719756000000, "updated_at": 1719756000000 }],
  "moods": { "2026-06-30": "😊", "2026-06-29": "😐" },
  "habit_defs": [{ "id": "ghi789", "name": "跑步 3km", "icon": "🏃", "order": 0, "created_at": 1719756000000 }],
  "habits": { "2026-06-30": { "ghi789": true }, "2026-06-29": { "ghi789": false } }
}
```

- 导入时合并策略：同日期覆盖，同 ID 覆盖，新 ID 追加
- 导入后显示结果：新增 X 条，更新 Y 条

### 8.4 存储层实现注意事项

**habit_YYYY-MM-DD 读取**：切换日期时，直接读取一个 key（`habit_YYYY-MM-DD`），不存在则返回 `{}`，再与 `habit_def_*` 列表合并渲染。不要逐个 key 查询。

**习惯删除后的 orphan 数据处理**：删除 `habit_def_xxx` 后，历史 `habit_YYYY-MM-DD` 中的 `"xxx": true` 仍然保留。渲染时如果某个 habit_id 在当前 `habit_def_*` 中找不到定义，直接跳过不显示（不报错、不显示「已删除」）。月历绿点判定不受影响（只看 key 是否存在）。

**待办排序**：有截止日期的按 deadline 升序 → 无截止日期 → 已完成（灰掉划线）。

**统计聚合**：在 `mine` 页 `onShow` 中遍历计算：
- 总字数：遍历所有 `entry_*`，累加 `content.length`
- 累计天数：统计 `entry_*` 的 key 数量
- 习惯完成率（本月）：分母 = 当月已过天数 × 当前习惯定义数，分子 = 遍历当月 `habit_YYYY-MM-DD` 中 `true` 的总数。例如：6 月有 30 天，定义了 3 个习惯，分母 = 30×3 = 90，分子 = 实际打勾数

### 8.5 待办月历性能优化

月历页判断待办绿点时，**不逐天逐条遍历**，改为一次分桶：

1. 月历 `onShow` 时，一次性读取全部 `todo_*` key（通过 `wx.getStorageInfoSync().keys.filter(k => k.startsWith('todo_'))`）
2. 遍历一次，将每条待办按 `deadline` 所在日期分桶到 `Map<dateStr, true>`
3. 月历渲染每天绿点时，直接查 Map，O(1)

500 条待办分桶一次 < 5ms，远优于 31×500 逐天遍历。

### 8.6 边界规则

| 场景 | 处理方式 |
|------|------|
| 存储接近上限 | 每次保存后调用 `wx.getStorageInfoSync()`，已用空间 > 8MB 时 Toast 警告「存储空间不足，请导出备份后清理」 |
| 导入 ID 冲突 | 同 ID 覆盖（最后写入者胜），不同 ID 追加，无需用户选择 |
| 导入前体积检查 | 导入前估算合并后总大小，超过 9MB 拒绝导入并提示「数据过大，请先清理历史数据」 |
| 分享功能 | 详情页右上角分享 = 生成纯文本摘要（日期 + 内容前 200 字 + 心情 + 习惯完成情况），通过微信「转发给朋友」发送 |
| 习惯定义上限 | 最多 20 个习惯定义，超出提示「习惯太多不易坚持，请先归档不需要的」 |
| 单条备忘/总结字数 | 不做硬限制，但超过 5000 字时 Toast 提示「内容较长，建议拆分」 |
| 存储用量可视化 | 「我的」页设置区域展示当前存储用量进度条（X MB / 10 MB） |
| 导出到微信收藏 | 导出 JSON 时除了「分享给朋友」外，提示用户「长按可收藏到微信」作为备份手段 |
| 待办推送提醒 | **不做**（服务通知需用户主动订阅 + 后端配合，MVP 不值得） |
| 按年归档清理 | 「我的」→「归档旧数据」入口：选择年份 → 导出该年 JSON 到微信文件 → 二次确认后删除本地该年数据，释放空间。纯手动触发，不做自动检测弹窗 |
| 导入冲突已知限制 | 同 ID 覆盖 = 最后写入者胜。若用户在两台设备分别修改同一条目再导入，会丢失一方修改。MVP 已知限制，不做冲突合并 |
| 定期导出提醒 | 每次打开「我的」页时，若距上次导出 > 30 天且数据量 > 3MB，顶部出现轻提示「建议导出备份，防止数据丢失」 |
